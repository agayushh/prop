Abstract: 	

Terminal buffers in Neovim lose all scrollback content when a session ends. While :mksession can reopen term:// URIs, the output from previously opened terminal is gone users see a blank shell instead of their previous work. This project focuses on the implementation of a persistent terminal scrollback by saving terminal buffer state to `stdpath(“state”)/term/`  using msgpack, and restoring it into real terminal buffers via nvim_open_term()+nvim_chan_send(). 
The restored content lives in proper libvterm scrollback not a normal buffer so users can scroll, search, and yank exactly as with a live terminal.

Phase 2 moves the serialization to C for full-fidelity color/attribute preservation by reading VTermScreenCell data directly from sb_buffer[].

When developers use the :terminal feature for things like seeing build results, running tests, checking logs, or working with SSH and REPLs, losing that terminal context when Neovim restarts disrupts the developer workflow. It means they have to go back and run those commands all over again and try to remember exactly how their workflow was set up. 
"Similar to undo/swap/backup directories, we might want a ~/.local/state/nvim/term/ directory. Could be useful for re-implementing :shell." – #28297
What cannot be restored?
The process itself — the underlying PTY job is gone. We restore the visual state, not the running process.
Environment variables, sub-shells, virtualenvs — these require plugin-level handling. Our job is to provide the infrastructure.

Phase 1 Approach (Lua-level MVP)
During :mksession, for each active terminal buffer in Neovim, I will extract all visible lines using nvim_buf_get_lines(buf, 0, -1, false). This captures the textual content currently displayed, including scrollback mirrored into the buffer. These lines will be bundled with relevant metadata such as terminal URI, working directory, dimensions (cols/rows), and a timestamp into a Lua table.
The data will then be serialized using vim.mpack.encode(), leveraging msgpack (which is already native to Neovim via ShaDa and RPC). The serialized output will be written to stdpath("state")/term/<hash>.mpack. Additionally, a Lua restore hook (e.g., vim.terminal._restore_scrollback(...)) will be injected into the generated :Session.vim, ensuring that restoration is triggered automatically when the session is sourced.
On session restore (nvim -S Session.vim), Neovim will first recreate terminal buffers using the existing term:// mechanism, which spawns fresh shell instances. Immediately after, the injected Lua restore function will execute: it reads the stored msgpack file, decodes it, and initializes a terminal instance using nvim_open_term(buf, {}). The saved output is then replayed via nvim_chan_send(chan, data), allowing the content to pass through libvterm and populate the internal scrollback buffer (sb_buffer[]). This ensures that the restored buffer behaves like a real terminal, supporting scrolling, searching (/), and yanking (y).
To improve reliability, an automatic save mechanism will be implemented using a TermClose autocmd, ensuring scrollback is persisted even if the user does not explicitly run :mksession, similar to how swap/undo files work.
Scope of Phase 1:
Preserved: text content, terminal dimensions, working directory, and term:// URI
Not yet preserved: colors, text attributes (bold/italic/underline), and cursor position (planned for Phase 2)

Phase 2 Approach

 In this phase, instead of relying on nvim_buf_get_lines() (which only captures plain text), the implementation will directly access the terminal scrollback stored inside libvterm at the C level. The scrollback is maintained in the sb_buffer[], which is an array of ScrollbackLine structures. Each ScrollbackLine contains multiple VTermScreenCell entries representing individual cells on the terminal screen.
Each VTermScreenCell stores detailed information about a character, including:
The character itself (schar)
Width (to support wide/CJK characters)
Styling attributes (bold, italic, underline, strikethrough, reverse, blink)
Foreground and background colors (VTermColor, supporting both indexed and RGB formats)
During the save process, the implementation will iterate through sb_buffer[0..sb_current-1], extracting each cell’s data and serializing it into a structured msgpack format. This data will be written to stdpath("state")/term/<hash>.mpack, replacing the plain-text representation used in Phase 1.
On restoration, instead of directly inserting text, the saved cell data will be converted into an ANSI escape sequence stream. For each row, the system will iterate through cells, track differences in attributes between consecutive cells, and emit appropriate ANSI SGR sequences (e.g., \033[1m for bold, \033[38;2;R;G;Bm for RGB foreground colors). Characters will then be emitted while correctly handling wide and multi-byte characters, and each row will be terminated with \r\n.
This reconstructed ANSI stream will be passed through nvim_chan_send(), allowing it to flow through libvterm. As a result, the terminal emulator reconstructs the scrollback exactly as it originally appeared, including colors, formatting, and layout.
Scope of Phase 2:
Preserved: text content, colors, text attributes (bold, italic, underline, etc.), wide characters, and overall visual fidelity
 Not yet preserved (optional extensions): exact cursor position and interactive shell state (process-level restoration)

Why nvim_open_term() + nvim_chan_send() instead of writing lines into a buffer?
This approach is necessary because it is the only way to restore content into a real terminal scrollback in Neovim. A terminal buffer in Neovim is fundamentally different from a normal buffer, as it is backed by libvterm.
The function nvim_open_term() creates a virtual terminal instance that is not connected to an external process but still behaves like a terminal emulator. Data can then be fed into this terminal using nvim_chan_send(), which sends input directly into the libvterm engine.
As the data flows through libvterm, it is interpreted as real terminal input, updating the screen and populating the internal scrollback buffer (sb_buffer[]). This process preserves terminal semantics such as ANSI color rendering, cursor movement, and overall terminal behavior.
In contrast, directly writing lines using nvim_buf_set_lines() only inserts plain text into a regular buffer (e.g., buftype=nofile). This approach cannot reproduce terminal behavior and results in a static view that lacks proper scrollback, formatting, and interactivity.
Therefore, using nvim_open_term() together with nvim_chan_send() ensures that the restored content behaves identically to a real terminal, supporting scrolling, searching, and yanking while maintaining full terminal semantics.


4. Storage format

Location: stdpath("state")/term/ as mentioned in the discussion

Format: msgpack native to Neovim


Schema:
`{
  version: 1,
  uri: "term:///home/user//12345:bash",  -- original term:// buffer name
  cwd: "/home/user/project",
  cols: 80,
  rows: 24,
  timestamp: 1711800000,
  lines: ["$ make", "Building...", "Done.", ...],
  -- Phase 2 additions:
  cells: [                    -- VTermScreenCell data per row
    [{char: "B", width: 1, bold: true, fg: [255,0,0], bg: [-1,-1,-1]}, ...],
    ...
  ]
}`
